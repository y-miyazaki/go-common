## Input

- Natural language request describing the topic/purpose (required)
- `document_type`: one of `readme`, `specification`, `architecture`, `design`, `design-decisions`, `troubleshooting`, `tutorial`, `general`, `module-catalog`, `monitoring`, `performance`, `security-coverage`, `maintenance-notes`, `improvements`, `other` (infer from request using [references/category-diataxis-mapping.md](references/category-diataxis-mapping.md); resolution rules in Workflow step 2)
- `profile`: `default`, `go`, or `terraform` (defaults to `default`; affects template selection for `specification` type only)
- `target_file`: path to target `.md` file (optional; required for `other` and `general` types)

## Output Specification

Return structured report per [references/common-output-format.md](references/common-output-format.md) with changed file paths and duplicate-check result. Regenerate `docs/index.md` only when files under `docs/` are created or updated.

## Execution Scope

- Directory layout follows [Diataxis](https://diataxis.fr/): `docs/tutorials/`, `docs/how-to/`, `docs/reference/`, `docs/explanation/`. Each `document_type` maps to one directory per [references/category-diataxis-mapping.md](references/category-diataxis-mapping.md).
- `readme` type MUST produce `README.md` at repository root.
- `general` is for creating new documents outside predefined types (template-driven). Requires explicit `target_file`.
- `other` is for updating existing documents outside predefined types (structure-preserving). Requires explicit `target_file`.

| | `general` | `other` |
|---|---|---|
| File state | Does not exist yet | Already exists |
| Template | Uses `general` template | Preserves original structure |
| Sections | Populated from template | Only improved, not reorganized |

- Do not rename/delete files, add YAML frontmatter, or run markdown linting (use markdown-validation skill).

### USE FOR:

- create or update docs under `docs/` or `README.md`
- update human-authored documents (e.g. "Improve the existing CONTRIBUTING.md") â€” a document is human-authored if it has no generation script or tool that produces it
- apply templates to specification, architecture, design, troubleshooting, tutorial docs

### DO NOT USE FOR:

- source code comments or docstrings
- non-markdown assets
- markdown linting or link checking (use markdown-validation skill)
- auto-generated API references (e.g., godoc, typedoc, terraform-docs output)
- files produced by a generation script (e.g., README.md generated by `terraform-docs`)

## Workflow

1. List markdown files in `docs/` (including subdirectories `tutorials/`, `how-to/`, `reference/`, `explanation/`). If `docs/` does not exist, create the directory structure and proceed.
2. Resolve `document_type`: use explicit value if present; otherwise infer from [references/category-diataxis-mapping.md](references/category-diataxis-mapping.md). If `target_file` is an existing file and type is unspecified, default to `other`. If ambiguous, ask user.
3. Resolve `target_file`: use directory from [references/category-diataxis-mapping.md](references/category-diataxis-mapping.md) + `{document_type}.md`; `readme` â†’ `README.md`; `general` and `other` require explicit `target_file`. If unresolved, ask user.
4. Gather context from source code:

   | document_type | Read |
   |---|---|
   | `readme` | directory structure, package config (go.mod, package.json, apm.yml), existing docs |
   | `specification` | tests, config files, CI workflows, existing behavioral contracts |
   | `architecture` | main package structure, entry points, key interfaces, dependency graph |
   | `design` | target module source, interfaces, internal dependencies |
   | `design-decisions` | git log for recent major changes, PR descriptions, existing ADRs |
   | `troubleshooting` | CI logs, error patterns, existing issue templates |
   | `tutorial` | quickstart scripts, example configs, test fixtures |
   | other types | files most relevant to the document's stated purpose |

   Context is sufficient when at least 2 sections can be populated with project-specific facts. If fewer than 2 sections have concrete source material, ask user for clarification. Otherwise, proceed and omit unpopulatable sections.
5. Select template: `category-templates-common-{document_type}.md`. For `specification` with `go` or `terraform` profile, use `category-templates-{profile}-specification.md`. If the resolved template file is missing, fall back to `category-templates-common-general.md` and record the fallback in the report.
6. Create or update the document:
   - For new documents: use template structure and populate sections using gathered context. Answer each section's guide comments before writing prose.
   - For `other` type (existing document update):
     - Preserve original structure.
     - Format fixes (code block languages, broken links) are applied automatically.
     - Content additions go at the end of existing sections.
     - Structural changes (reordering, splitting, removing sections) require user confirmation.
   - Remove sections that cannot be populated with concrete information rather than leaving generic filler.
7. Run case-insensitive duplicate check (skip for `other` type). A duplicate exists when another file in `docs/` has the same basename regardless of subdirectory or case (e.g., `reference/design.md` duplicates `explanation/design.md`). Fail on duplicates.
8. Regenerate `docs/index.md` if files under `docs/` changed (use index template from common-checklist).
9. IF README has docs-index markers (`<!-- docs-index-start -->` / `<!-- docs-index-end -->`), update content inside markers. IF markers are malformed or ambiguous, skip marker update and note in report. IF no markers exist, skip.
10. Return report per [references/common-output-format.md](references/common-output-format.md).

### Error Handling

| Condition | Severity | Action |
|---|---|---|
| `docs/` does not exist | Recoverable | Create directory structure, continue |
| Template file missing | Recoverable | Fall back to `general` template, note in report |
| Duplicate filename detected | Fatal | Stop, report NC-01 failure |
| `document_type` ambiguous | Blocking | Ask user, do not guess |
| Reference file (`common-checklist.md`, `category-diataxis-mapping.md`) missing | Fatal | Stop, report missing dependency |
| Context gathering yields 0 populatable sections | Blocking | Ask user for input |
| README marker malformed | Recoverable | Skip marker update, note in report |
