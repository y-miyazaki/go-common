## Terraform Specification Template Variant

This file provides a profile-specific override for `document_type=specification`.
For other `document_type` values, use the corresponding `category-templates-common-*.md` templates.

## specification_terraform

```markdown
# Terraform Specification

This document defines repository behavior, infrastructure intent,
environment-specific configuration, and operational expectations
for Terraform-managed resources.

Do not duplicate autogenerated terraform-docs outputs.
Focus on architectural intent, operational risk, lifecycle behavior,
security boundaries, and environment-specific considerations.

## Documentation Priorities

Prioritize:
- infrastructure intent
- operational behavior
- state and lifecycle implications
- replacement impact
- security boundaries
- environment-specific behavior
- failure recovery expectations

Deprioritize:
- provider-default explanations
- obvious Terraform syntax
- autogenerated variable/resource tables
- duplicated terraform-docs output

Avoid:
- describing every resource equally
- embedding account-specific secrets or identifiers
- repeating provider documentation
- omitting destructive-change warnings
- hiding state-sensitive behavior

## Scope

| Item           | Detail                                |
| -------------- | ------------------------------------- |
| Stacks         | <comma-separated list of stack paths> |
| Environments   | <e.g., dev, staging, production>      |
| Cloud Provider | <e.g., AWS, GCP>                      |
| Exclusions     | <what is explicitly out of scope>     |

Out-of-scope items:

- Module input/output variable documentation (maintained by terraform-docs)
- Resource specification tables (maintained by terraform-docs outputs)
- Provider usage tutorials
- Basic Terraform language explanations
- <other exclusions>

## Infrastructure Intent

Document:
- what infrastructure problem is solved
- intended operational scope
- whether stacks are reusable or environment-specific
- architectural assumptions and constraints
- external systems or dependencies

Consider:
- composition vs standalone usage
- multi-account or multi-region assumptions
- shared infrastructure boundaries

## Resource and Lifecycle Design

Describe:
- major infrastructure components
- resource relationships and dependencies
- lifecycle-sensitive resources
- resources sensitive to recreation
- stateful or drift-sensitive resources

Document:
- destructive replacement risks
- import expectations
- immutable infrastructure assumptions
- zero-downtime limitations
- ordering or dependency constraints

Avoid:
- listing every resource without rationale

## Environment Strategy

Document:
- environment isolation strategy
- naming conventions
- backend/state separation strategy
- differences between environments
- feature toggles or conditional infrastructure

Consider:
- promotion workflow between environments
- blast radius containment
- shared vs isolated resources

## Security Considerations

Always evaluate:
- IAM/RBAC boundaries
- least-privilege assumptions
- public exposure risks
- secret handling
- encryption requirements
- network boundaries
- cross-account access
- auditability expectations

Document:
- sensitive resources
- trust relationships
- externally accessible components
- compliance-sensitive behavior

## Operational Characteristics

Document:
- monitoring expectations
- alerting assumptions
- failure recovery behavior
- scaling characteristics
- cost-sensitive resources
- maintenance constraints

Consider:
- disaster recovery assumptions
- operational ownership
- expected deployment frequency
- runtime dependencies

## Validation and Safety Checks

Required validation examples:

- `terraform fmt -check`
- `terraform validate`
- `terraform plan`
- `tflint --init`
- `tflint --recursive`
- `<project specific checks>`

Document:
- CI/CD validation flow
- policy enforcement tools
- drift detection strategy
- approval requirements
- plan review expectations

## Change Management

Define:
- plan/apply workflow
- approval requirements
- rollback limitations
- emergency change procedures
- state migration expectations

Document:
- resources requiring extra review
- destructive change approval rules
- operational freeze requirements
- expected deployment sequencing

## Decision Prompts

Consider:
- Which resources are state-sensitive?
- Which variables can trigger replacement?
- Which resources are externally referenced?
- Which infrastructure changes are irreversible?
- Which operations require downtime?
- Is least privilege maintained?
- What is the blast radius of failure?
- Which components are hardest to recover?
- Are dependencies explicitly documented?
- Can drift cause operational issues?

## Related Documentation

Reference related documents when applicable:

- architecture overview
- environment strategy
- security coverage
- operational runbooks
- disaster recovery procedures
- module catalog
- monitoring guidance
```
