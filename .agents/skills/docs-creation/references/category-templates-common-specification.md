## specification

Templates are guidance and review rubrics, not rigid prose generators.
Adapt structure and depth to the repository and context.
Replace all `<placeholder>` values with actual content.
Remove sections that do not apply rather than leaving them empty.

```markdown
# <Project Name> Functional Specification

This document defines behavioral guarantees, operational expectations,
configuration behavior, and implementation constraints for <project>.

Focus on:
- intended behavior
- feature boundaries
- lifecycle expectations
- compatibility assumptions
- operational impact

Avoid:
- implementation trivia
- autogenerated API/resource listings
- framework/provider tutorials

## Scope

| Item         | Detail                             |
| ------------ | ---------------------------------- |
| Features     | <comma-separated list of features> |
| Environments | <e.g., dev, staging, production>   |
| Exclusions   | <what is explicitly out of scope>  |

Out-of-scope items:

- <explicit exclusions>
- autogenerated documentation
- basic language/framework explanations

## System Responsibilities

Describe:
- primary responsibilities
- ownership boundaries
- intended consumers
- external dependencies

## Behavioral Guarantees

Document:
- expected behavior
- invariants
- state transitions
- compatibility expectations
- failure behavior

## Operational Characteristics

Document:
- deployment/runtime assumptions
- scaling expectations
- operational dependencies
- maintenance expectations
- observability assumptions

## Configuration and Defaults (Optional)

| Parameter | Default   | Source            | Notes   |
| --------- | --------- | ----------------- | ------- |
| `<name>`  | `<value>` | `<env/file/flag>` | <notes> |

Document:
- dangerous configuration
- override precedence
- environment-specific behavior

## Validation and Safety Checks

- `<lint or format check>`
- `<test command>`
- `<security or policy check>`
- `<project specific checks>`

## Decision Prompts

Consider:
- What behavior must remain stable?
- What assumptions exist around runtime behavior?
- Which failures are recoverable?
- Which changes are operationally risky?
- Which boundaries are externally consumed?
```
